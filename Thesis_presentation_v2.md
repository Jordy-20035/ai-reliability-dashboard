## Slide 2 — Проблема: data drift и риски в продакшене

**4-card layout (numbered 1–4):**

1. **Data drift** — Сдвиг распределений входных данных относительно
   распределений при обучении.

2. **Скрытый риск** — Снижение качества модели «тихо», без явных ошибок
   в коде сервиса.

3. **Недостаточность детекторов** — Нужен аудит, переобучение,
   версионирование и деплой, а не просто алерт.

4. **Фрагментация инструментов** — Evidently / NannyML / MLflow / Airflow
   решают каждый свою часть; нужна интеграционная связка.

набора разрозненных MLOps-инструментов.

## Slide 4 — Статистические методы + Архитектура

**Header:** Три метода детектирования + 5 модулей платформы

**Left (методы, 3 короткие колонки):**

- **PSI** = Σ (Qᵢ − Pᵢ) · ln(Qᵢ / Pᵢ); 10 бинов; бэнды < 0.10 / 0.10–0.25 / ≥ 0.25.
- **KS** = supₓ |Fₙ(x) − Gₘ(x)|; α = 0.05; численные признаки.
- **χ²** = Σ (Oᵢ − Eᵢ)² / Eᵢ; α = 0.05; категориальные (rare → `__other__`).

**Right (архитектура):**

```
┌───────────────────────────────────────────────────────────┐
│  REST API (FastAPI)             SPA Dashboard (React/MUI) │
└─────────────────────────┬─────────────────────────────────┘
       ┌──────────────────┼──────────────────┐
┌──────▼──────┐  ┌────────▼────────┐  ┌──────▼──────┐
│   drift_    │  │  orchestration  │  │  retraining │
│  detection  │  │  Engine+Policy+ │  │  pipeline + │
│  (PSI/KS/χ²)│  │  Actions+Store  │  │  promotion  │
└─────────────┘  └────────┬────────┘  └──────┬──────┘
                 ┌────────▼──────────────────▼────────┐
                 │  lifecycle      data_management    │

```

**Стек:** Python 3.11 + FastAPI + scikit-learn + SQLite; React + MUI; Docker Compose.

## Slide 5 — Политика срабатывания и жизненный цикл моделей

**Header:** DriftThresholdPolicy + Lifecycle

**Пороги политики:**

| Счётчик                       | Порог | Логика                              |
|-------------------------------|:-----:|-------------------------------------|
| n_features_high_psi           | > 0   | Любая «high»-PSI фича → триггер     |
| n_numeric_ks_significant      | > 2   | Допускаем ~2 KS-FP при α=0.05       |
| n_categorical_chi2_significant| > 3   | Допускаем ~3 χ²-FP при α=0.05       |

**Action chain при триггере:** LogAction → WebhookAlertAction → RetrainPipelineAction.
Каждый запуск сохраняется в `artifacts/orchestration.db` (timestamp, scenario, причины, JSON-сводка).

**Жизненный цикл:** `development → staging → production → archived`.

**Правило продвижения:** новая модель становится `production` ⇔
**macro-F1 ≥ macro-F1 чемпиона**. Иначе остаётся в `development`
с пометкой `f1_macro_below_champion`; production-указатель **не меняется**.
**Безопасность:** production-модель не ухудшается по выбранной метрике.

## Slide 6 — Эксперименты: Adult + Credit Card Fraud

**Adult Census Income** (48 842 записи, 14 признаков, positive class ≈ 24%):

| Сценарий       | n_high_psi | n_ks_sig | n_chi2_sig | Триггер |
|----------------|:----------:|:--------:|:----------:|:-------:|
| random_holdout |      0     |   0–2    |    0–1     |   нет   |
| age_shift      |      4     |    5     |     5      |   да    |

**Credit Card Fraud** (D1/D2/D3 — разбиение по `Time`, 29 численных признаков):

| Сценарий          | n_high_psi | n_ks_sig | Средний PSI | Триггер |
|-------------------|:----------:|:--------:|:-----------:|:-------:|
| fraud_d1_vs_d2    |      3     |     7    |    0.11     |   да    |
| fraud_d2_vs_d3    |      5     |     9    |    0.16     |   да    |
| fraud_d1_vs_d3    |      8     |    12    |    0.23     |   да    |

**Наблюдение:** на `random_holdout` политика не срабатывает (счётчики
в пределах ожидаемого статистического шума при α=0.05); на `age_shift`
и всех temporal-сценариях fraud — срабатывает. Дрейф монотонно растёт
с увеличением темпорального расстояния.

## Slide 7 — Результаты переобучения + демонстрация дашборда

**Сравнение моделей (Credit Card Fraud) — до и после переобучения:**

| Обучение            | Тест | ROC-AUC | **PR-AUC** | Recall |
|---------------------|:----:|:-------:|:----------:|:------:|
| D1 only             |  D2  |  0.884  |  **0.668** | 0.784  |
| D1 only (baseline)  |  D3  |  0.949  |  **0.609** | 0.820  |
| **D1+D2 (adapted)** |  D3  |  0.978  | **🟢 0.746** | 0.877  |

**Ключевой результат:** переобучение на свежих данных (D2) увеличило
PR-AUC на тестовом периоде D3 с **0.609 → 0.746 (+0.137)**.

4. **Inference** — playground: выбор профиля (`adult`/`fraud`), JSON-ввод,

## Slide 8 — Вклад, ограничения, развитие + Спасибо

**Вклад работы**
- Интегрированный MLOps-контур: **drift → decision → retrain → lifecycle → inference** в одной системе.
- Реализация для двух разных доменов (Adult + Fraud temporal).
- Прослеживаемость: provenance связывает датасет, baseline, эксперимент, версию модели.
- Открытая, воспроизводимая платформа (Docker Compose, SQLite).

**Ограничения**
- Демо-датасеты и эвристические пороги принятия решений.
- Отсутствие enterprise-аутентификации (RBAC, SSO).
- Single-host развёртывание (Docker Compose, не Kubernetes).

**Дальнейшее развитие**
- Quality gates при переводе в production (A/B tests, shadow / canary deploy).
- Расширение комбинаций переобучения (D1+D3, D2+D3) и обучаемые политики.
- Оповещения в Slack / email; дрейф эмбеддингов для не-табличных данных.
