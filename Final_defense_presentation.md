## Slide 3 — Методология детектирования + Архитектура платформы

**Header:** PSI / KS / χ² + 5 модулей в единой системе

**Top — три метода в 3 колонках:**

**PSI (Population Stability Index)**
- PSI = Σ (Qᵢ − Pᵢ) · ln(Qᵢ / Pᵢ)
- 10 квантильных бинов
- Бэнды: < 0.10 норма · 0.10–0.25 умеренный · ≥ 0.25 высокий

**KS (Колмогоров–Смирнов)**
- KS = supₓ |Fₙ(x) − Gₘ(x)|
- α = 0.05
- Применяется для численных признаков

**χ² (Хи-квадрат)**
- χ² = Σ (Oᵢ − Eᵢ)² / Eᵢ
- α = 0.05
- Категориальные признаки · редкие категории → `__other__`

**Bottom — схема архитектуры:**

```
┌────────────────────────────────────────────────────────────┐
│  REST API (FastAPI)              SPA Dashboard (React/MUI) │
└──────────────────────────┬─────────────────────────────────┘
       ┌──────────────────┼──────────────────┐
┌──────▼──────┐  ┌────────▼────────┐  ┌──────▼──────┐
│   drift_    │  │  orchestration  │  │  retraining │
│  detection  │  │  Engine+Policy+ │  │  pipeline + │
```

## Slide 4 — Политика срабатывания и жизненный цикл моделей

**Header:** DriftThresholdPolicy + Lifecycle

**Left — пороги политики:**

| Счётчик                         | Порог | Логика                                |
|---------------------------------|:-----:|---------------------------------------|
| n_features_high_psi             | > 0   | Любая «high»-PSI фича → триггер       |
| n_numeric_ks_significant        | > 2   | Допускаем ~2 KS-FP при α=0.05         |
| n_categorical_chi2_significant  | > 3   | Допускаем ~3 χ²-FP при α=0.05         |

**Action chain (при триггере):**

1. **LogAction** — запись события в `artifacts/orchestration.db`.
2. **WebhookAlertAction** — POST на внешний webhook (опц.).
3. **RetrainPipelineAction** — запуск пайплайна переобучения.

**Right — жизненный цикл моделей:**

```
Development → Staging → Production → Archived
                              ▲
                              │ auto-promote
                              │
                  macro-F1 новой ≥ macro-F1 чемпиона
```

**Правило продвижения:** новая модель становится `production` **только если**
macro-F1 новой ≥ macro-F1 чемпиона. Иначе — статус `development`
с пометкой `f1_macro_below_champion`; production-указатель не меняется.

**Highlighted bottom bar:**
**Гарантия безопасности** — production-модель никогда не ухудшается
по выбранной метрике.
